# BGP Scan Visualizer

## Overview

This project is a web-based visualizer for exploring inferred network topology generated from BGP scan data. It was developed as part of a project on **inferring BGP scan data**. More information about that project will be added once it is publicly available.

The application uses a Neo4j and a PostgreSQL database as its backend and provides an interactive graph visualization with support for inspecting neighbouring entities, executing read-only Cypher queries, and browsing the inferred topology.

A deployed instance of the visualizer is available at:

<https://wesselritskes.github.io/BGPScanVisualizer/>

The application is deployed with the following services:
* **AuraDB**, for hosting the Neo4j database.
* **Neon**, for hosting the Credentials database.
* **Render**, for hosting the back-end.
* **GitHub Pages**, for hosting the front-end.

Access to the application requires valid login credentials. See the **Managing User Credentials** Section for more details. To gain access to the deployed instance contact Wessel Ritskes, w.r.ritskes@student.utwente.nl.

## Deployment

### Environment Variables

The application requires the following environment variables:

| Variable          | Description                                                             |
|-------------------|-------------------------------------------------------------------------|
| `ADMIN_KEY_HASH`  | Argon2 hash of the administrator key used to manage user credentials.   |
| `USERNAME_SECRET` | Secret key used for HMAC hashing usernames before storage.              |
| `NEO4J_USERNAME`  | Neo4j database username.                                                |
| `NEO4J_PASSWORD`  | Neo4j database password.                                                |
| `NEO4J_URI`       | URI of the Neo4j database instance.                                     |
| `DB_URI`          | URI of the PostgreSQL database instance that will hold the credentials. |

### Generating Authentication Keys

The `ADMIN_KEY_HASH` and `USERNAME_SECRET` values can be generated using:

```bash
python generate_keys.py
```

The script prints:

* an **administrator key**
* its corresponding **Argon2 hash**
* a randomly generated **USERNAME_SECRET**

Store the **hash** as `ADMIN_KEY_HASH` in your environment.

Keep the **administrator key** in a secure location, as it is required later for managing user accounts.

The Neo4j and PostgreSQL environment variables depend on your own Neo4j deployment.

## Running the Application

Start the FastAPI server with:

```bash
uvicorn neo4j_visualizer:app --host 0.0.0.0 --port 8000
```

If deploying on a platform that provides a `PORT` environment variable (such as Render), use:

```bash
uvicorn neo4j_visualizer:app --host 0.0.0.0 --port $PORT
```

## Managing User Credentials

Before users can log in, credentials must be generated.

The included `manage_credentials.py` script communicates with the running application to create or remove user accounts.

Before running the script:

1. Verify that the server IP or URL configured in the script points to your deployment.
2. Use the **administrator key** generated by `generate_keys.py` (not the stored hash).

The script provides three options:

* `gen` – Generate credentials for a username.
* `del` – Delete the credentials associated with a username.
* `quit` – Exit the program.

When generating credentials:

* If the username does not exist, a new account is created.
* If the username already exists, its password is regenerated and the previous password becomes invalid.
* The newly generated password is displayed once and should be communicated securely to the user.

Deleting credentials removes the account completely.

## Using the Application

After obtaining valid credentials:

1. Open the deployed application in your browser.
2. Log in using the provided username and password.
3. Explore the graph by clicking nodes or by executing read-only Cypher queries.

Additional information, example queries, and usage instructions are available directly within the application by clicking the **?** (Help) button in the top-right corner.

---

## Security Notes

* Usernames are not stored in plaintext. Instead, they are stored as HMAC-SHA256 hashes using `USERNAME_SECRET`.
* Passwords are stored using Argon2id hashes.
* Authentication uses secure session cookies after successful login.
* The administrator key is never stored in plaintext by the server; only its Argon2 hash is kept in the environment.
